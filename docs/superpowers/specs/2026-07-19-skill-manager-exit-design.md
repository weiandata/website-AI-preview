# Skill 管理器退出按钮设计

日期：2026-07-19

## 背景

管理器由 `启动Skill管理器.command` 拉起，服务跑在终端窗口里。关闭浏览器标签页不会停掉服务——管理员通常以为已经退出了，实际进程还在后台运行。今天就出现过这个后果：一个已保存但未发布的 Skill 一直留在本机，而管理员看不到任何提示。

管理员是非程序员，不应该需要回到终端按 Control-C 才能收工。

## 目标

在界面里提供一个真正的退出入口，并在退出前把「还没了结的事」摆到台面上。

不做的事：不做定时自动退出，不做会话恢复，不做多窗口协调。管理器仍然是本机单人工具。

## 界面

退出按钮放在顶栏最右侧，与导入/导出一组之间留一条分隔线。

不放在底栏：底栏紧挨着「保存并发布」，误点的代价是服务真的停掉，管理员得回终端重新启动。

点击后先取一次退出前检查，再弹出确认窗口，列出三类未了结的事：

| 检查项 | 数据来源 |
| --- | --- |
| 未保存的编辑 | 页面里的待保存队列 |
| 已保存但未发布 | `git status` 中未提交的 `content/skills/*.md` |
| 上次 push 失败挂着的提交 | `GitPublisher` 中等待重试的本地提交 |

窗口提供「返回处理」（默认、主按钮）与「仍然退出」两个选择。三类都干净时窗口只说明一切已发布，保留一个确认键——干净时也要确认，是因为误点的代价是服务停止，多一次点击换一个不会手滑的保险。

确认退出后，页面替换为告别页，说明服务已停止、可以关闭页面、下次如何重新打开。

## 服务端

新增两个接口：

- `GET /api/exit/preview` 返回 `{ unpublishedPaths, pendingPush }`。
- `POST /api/exit` 先回应，再停止进程。

退出动作做成可注入的钩子 `onExit`，默认实现关闭监听并退出进程。测试注入替身，因此不会杀掉测试进程。

未发布清单复用 `GitPublisher.pendingContentPaths()`，与发布流程用的是同一份判断，两处不会各说各话。

## 测试

服务端：检查接口如实报告未发布文件与待 push 提交；退出接口先响应再触发钩子。

界面：确认窗口逐条列出未保存与未发布内容；「仍然退出」调用接口并换成告别页；「返回处理」关闭窗口且不调用接口。
