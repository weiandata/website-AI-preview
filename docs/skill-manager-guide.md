# Skill 管理器使用说明

这份说明写给负责维护 Skill 内容的管理员，不需要任何编程基础。

Skill 管理器只在你这台 Mac 上运行。它不是网站的一部分，别人无法从互联网打开它。

---

## 1. 启动与退出

在项目文件夹里双击：

```text
启动Skill管理器.command
```

启动器会自己切换到项目目录，所以不管你当前在哪个文件夹双击，它都能正确运行。

第一次启动时会自动安装依赖，需要等待几分钟。之后启动只需要几秒。

准备好之后浏览器会自动打开：

```text
http://127.0.0.1:4174
```

如果提示找不到 Node.js 或版本过低，请先安装 Node.js 20 或更高版本，再重新双击。

### 收工时点右上角的「退出」

**只关掉浏览器标签页是不够的**——管理器服务还在后台跑着，你会以为已经退出了。

点顶栏右上角的「退出」，它会先检查还有没有没了结的事，再问你要不要真的退出：

| 检查项 | 说明 |
| --- | --- |
| 还没保存的编辑 | 退出后会丢失，回到上次保存的样子 |
| 已保存但没发布的 Skill | 逐个列出文件名，内容还在你电脑上，只是网站上看不到 |
| 上次 push 失败挂着的提交 | 可以先点「重试发布」 |

窗口给两个选择：**返回处理**（回去把事情做完）和**仍然退出**。都干净时窗口会写「一切都已发布」。

确认退出后服务会真正停止，页面变成告别页，终端窗口那边会显示「已停止」。想再用时重新双击启动器即可。

如果不想用按钮，关闭终端窗口或按 `Control-C` 也一样能停，只是不会有退出前检查。

## 2. 界面分三栏

| 位置 | 内容 |
| --- | --- |
| 左栏 | 全部 Skill 列表、搜索、分类与状态筛选、新建/复制/删除、首页精选顺序 |
| 中栏 | 当前 Skill 的完整表单，所有字段都在同一页，不折叠 |
| 右栏 | 实时预览，随着你的输入立刻更新 |

底部一直显示还有多少项没保存。

---

## 3. 两个保存按钮的区别

这是最重要的一件事。

| 按钮 | 做什么 | 不做什么 |
| --- | --- | --- |
| **仅保存** | 把修改写进你这台电脑的 `content/skills` 文件夹 | 不联网，不碰 GitHub，网站不会更新 |
| **保存并发布** | 先做和「仅保存」一样的事，然后提交并推送到 GitHub | 不会碰 Skill 内容以外的任何文件 |

两个按钮都会先弹出确认窗口，列出这次会**新增、更新、改名、删除**哪些文件。看清楚再点确认。

保存完成后会显示每个文件的结果。如果某个文件失败了，它会留在待保存清单里，其他成功的文件不受影响，你修好后再保存一次即可。

保存时管理器会顺手把 Markdown 的排版问题（比如列表前少一个空行）自动修好，用的是和 GitHub 上那套检查完全相同的工具和配置。你不用管这件事，也不会看到任何提示。

**已经保存但还没发布的内容不会丢。** 就算你关掉管理器再重新打开，它依然记得哪些 Skill 存了盘却没发布，点「保存并发布」就能接着发。

## 4. 发布到网站

点「保存并发布」并确认保存后，管理器会检查一遍 GitHub 状态和这些文件里的所有链接，然后出现发布确认窗口，上面写着：

- 这次会发布哪几个 `.md` 文件
- 哪些修改**不会**被发布（比如程序代码文件，管理器绝不会碰它们）
- 发布说明（commit message），可以直接用默认的，也可以自己改

如果你之前已经点过「仅保存」，直接点「保存并发布」也能发——不需要先随便改点什么把按钮「激活」。

确认发布后，状态会依次显示：

```text
正在检查 GitHub
已提交，正在 push
GitHub 已接收，Cloudflare 正在发布
```

看到「GitHub 已接收，Cloudflare 正在发布」就说明本地这边做完了。Cloudflare 大约需要 1–3 分钟完成构建，管理器不会自动查询 Cloudflare 的进度，请稍后自己打开网站确认。

### 发布被拒绝怎么办

发布窗口如果显示「现在还不能发布」，按提示处理：

| 提示 | 原因 | 怎么办 |
| --- | --- | --- |
| 链接打不开（HTTP 404） | 某个网址失效了 | 窗口会指名是哪一条，回表单改掉再发 |
| 只能从 main 分支发布 | 当前不在 `main` 分支 | 找开发同事把仓库切回 `main` |
| GitHub 上的 main 比本机新 N 个提交 | 别人先改了内容 | 找开发同事同步（`git pull`）后再发布 |
| 仓库有未解决的冲突文件 | 上次同步没做完 | 找开发同事处理冲突 |
| origin 不是 GitHub 仓库 | 仓库配置不对 | 找开发同事检查 |

管理器**永远不会**强制推送，也不会丢掉别人的修改。它宁可拒绝，也不会覆盖。

### 关于链接检查

发布前，管理器会逐条打开你填的 GitHub 地址、官网地址、下载地址，以及正文里的链接。

- **打不开的**（比如 404）会拦住发布，并告诉你是哪一条。这类问题在 GitHub 上也会被检查出来，提前挡住省得事后返工。
- **只是没连上的**（网络慢、对方站点临时抽风）只提示，不拦你——网络不好不等于地址写错了。

### push 失败怎么办

如果显示「已保存，但 push 失败；可以重试发布」，通常是网络问题。

你的修改**已经安全地提交在本机**，没有丢。等网络恢复后点底部的「重试发布」即可。

## 5. 字段说明

| 字段 | 作用 |
| --- | --- |
| **状态** | `已发布` 才会出现在网站上；`草稿` 完全不会出现在任何公开页面、搜索、结构化数据和 sitemap 里 |
| **Slug** | 决定网址 `/skills/<slug>/`。改它会让旧网址失效，所以改已发布 Skill 的 slug 需要额外勾选确认 |
| **精选** | 勾上才会出现在首页精选区 |
| **精选顺序** | 数字越小越靠前。用左栏「首页精选顺序」面板拖动卡片，或用上下按钮调整，顺序会自动重新编号 |
| **已核验** | 显示一个核验标记，表示你亲自确认过这个 Skill |
| **中文/英文字段** | 两种语言并排填写，网站会按访客选择的语言显示 |

## 6. 导入和导出

- **导入 Markdown**：可以一次选多个 `.md` 文件。管理器会先检查，把它们分成「新增」「冲突」「无法解析」。冲突的必须逐个勾选「允许覆盖」才会替换。导入只加入编辑队列，**不会立刻写盘**，你还需要点保存。
- **导出当前**：把当前这个 Skill 存成一个 `.md` 文件。
- **导出全部**：把所有 Skill 加上官方模板打包成一个 zip，适合做备份。

一个 `.md` 文件就是一个完整的 Skill，可以直接备份、交给别人、或者以后再导入回来。

---

## 7. 删除与找回

点「删除」并确认后，这个 Skill 进入待删除队列；真正删除发生在你点保存之后。

删除的文件**不会被真正销毁**，而是移到项目里的：

```text
.skill-manager-trash/
```

需要找回时，把文件从这个文件夹移回 `content/skills/` 并去掉文件名里的时间后缀即可，或者找开发同事帮忙。

---

## 8. 内容出错了怎么办

如果某个字段填得不对（比如 slug 里有大写字母或空格），保存时会显示具体是哪个文件、哪个字段的问题。按提示改好再保存即可，错误的内容不会被写进去。

如果网站构建失败（Cloudflare 上看到红色的 Failed），说明有内容不符合格式。请把 Cloudflare 上的错误信息发给开发同事。

### GitHub 上出现红叉

在 GitHub 的提交页面，除了 Cloudflare 的部署状态，还有一项叫 **Repository checks** 的检查，负责校验仓库里所有 Markdown 的排版和链接是否有效。

它和网站是否上线**没有关系**——Cloudflare 那一项绿了，网站就是好的。

管理器在保存和发布时已经做过同样的两项检查，所以你发布的内容正常不会让它变红。如果还是红了，把链接发给开发同事看。

### 回滚到上一个版本

如果发布后发现内容有问题，最快的办法是在管理器里改回来再发布一次。

如果需要整体回退，请找开发同事在 GitHub 上操作：仓库的每一次发布都是一个独立 commit，可以精确地还原。

---

## 9. 底线保证

管理器在设计上做了这些限制，你不用担心误操作：

- 只会提交 `content/skills/<slug>.md` 这种内容文件，永远不碰程序代码
- 只提交你自己改过的 Skill 文件，一个一个地加，不会 `git add .`
- 永远不会强制推送（force push），不会 `git reset --hard`
- 远端有更新时宁可拒绝发布，也不会覆盖别人的修改
- 链接打不开时宁可拦住发布，也不会把死链发到网站上
- 删除是可恢复的
- 保存使用原子写入，中途断电也不会写出半个文件
- 退出前会告诉你还有什么没做完，但不会因此把你困住
